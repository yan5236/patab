# PaTab 代码风格

## 基本原则

- 最小改动，只改与任务直接相关的文件。
- 模块单一职责，避免上帝对象。
- 前端 UI 默认组件化开发，能复用的界面、样式和交互都抽象为 Vue 组件。
- 单文件原则上不超过 500 行；超过时优先拆分，但不为小改动做无关重构。
- 优先复用现有 helper、类型和组件。
- 函数级注释使用中文，说明职责、输入边界或关键副作用。
- 不随意改动已有中文注释。

## 组件化规范

- 页面级组件只做布局编排，不堆积具体 UI 细节。
- 可复用 UI 放到 `components/common/`，领域组件放到对应业务子目录。
- 重复出现两次的结构、样式或交互，优先抽成组件或 composable。
- 组件对外接口保持小而明确：props 输入、emit 输出、slot 扩展。
- 不为尚未出现的场景预留复杂抽象，先满足当前复用需求。
- 移动端主屏优先使用纵向流式布局，Dock 固定底部并允许横向滑动；桌面端保留固定坐标桌面布局。
- 移动端设置弹窗使用二级菜单，先展示设置分类，进入分类后再显示具体选项；内容滚动到返回按钮或分类标题不可见时，把它们动态提升到弹窗标题行。
- 移动端可拖拽图标不要直接使用方向性 `touch-action: pan-x/pan-y`；长按拖拽接管图标触摸，普通滑动由最近可滚动容器处理。
- 紧凑模式在移动端必须按实际渲染矩形计算拖拽插入位置；不要用桌面固定列数推断手机流式网格。
- 鼠标点击路径不能提前捕获 pointer；只有真正进入拖拽后才允许吞掉后续 click。

## 命名

- Vue 组件使用 PascalCase，例如 `ScreenViewport.vue`。
- composable 使用 `useXxx.ts`。
- store 使用业务名，例如 `launcher.ts`、`ui.ts`。
- 类型名使用 PascalCase。
- 常量使用全大写或语义化 `const`，按现有文件风格保持一致。

## 目录约定

- 页面结构组件放在 `components/screen/`。
- Dock 相关组件放在 `components/dock/`。
- 弹窗放在 `components/modals/`。
- 设置面板放在 `components/settings/`。
- 纯函数放在 `utils/`。
- 跨组件类型放在 `types/index.ts`。
- 测试放在 `src/__tests__/`。
- 设置页新增搜索、壁纸等领域面板时，优先放在 `components/settings/`，规则清洗放在对应 `utils/`。

## 注释规范

- 公共函数、复杂交互函数、持久化和兼容逻辑必须有中文函数级注释。
- 简单变量不写废话注释。
- 解释“为什么”优先于解释“代码逐行做了什么”。

## 错误处理

- 用户数据读取失败时安全回退，不抛到界面层。
- URL、壁纸、图标等外部资源失败时保留可操作界面。
- 业务 action 内保持状态一致，避免组件绕过 store 直接改数据。

## 测试规范

- 非平凡逻辑需要最小可运行测试。
- 新增持久化字段、拖拽规则、网格算法或 URL 处理时必须补测试。
- 优先测试 `utils/` 和 store 行为，少测样式细节。

## 提交前检查

```sh
cd patab-web
pnpm build
pnpm test:unit
```
